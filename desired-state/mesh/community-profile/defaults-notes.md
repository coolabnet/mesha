# defaults-notes.md ‚Äî Understanding the LibreMesh Configuration Hierarchy

## What `lime-community` does

`lime-community` is the community-wide configuration file for a LibreMesh mesh network. When the `lime-config` command runs on a node, it reads a layered set of configuration files in priority order and merges them to generate the final OpenWrt UCI configuration. The `lime-community` file sits in the middle of that stack and defines settings that should be the same across every node in the network.

The three main configuration files in the hierarchy are:

| File | Scope | Who edits it |
|------|-------|--------------|
| `lime-defaults` | LibreMesh software defaults | LibreMesh upstream (do not edit) |
| `lime-community` | Community-wide settings | Community maintainers via this repo |
| `lime-node` | Single-node overrides | Placed on the individual node only |

An additional auto-generated file, `lime-autogen`, is created by `lime-config` and must never be treated as a source of truth or edited by hand. It is regenerated on every `lime-config` run.

## How the files interact

When `lime-config` runs on a node, it:

1. Reads `lime-defaults` for baseline values.
2. Reads `lime-community` and applies community-level overrides on top.
3. Reads `lime-node` (if it exists on the node) and applies per-node overrides on top.
4. Generates `lime-autogen` with the fully merged configuration.
5. Applies `lime-autogen` to the actual OpenWrt UCI configuration.

This means that any setting in `lime-community` takes precedence over the same setting in `lime-defaults`, and any setting in `lime-node` takes precedence over `lime-community`.

For node-specific overrides managed through this repository (rather than placed directly on the node), see `desired-state/mesh/node-overrides/` and the `mesh-onboarding` skill documentation.

## Community-wide settings (belong in `lime-community`)

These settings should be the same on every node and belong in `lime-community`:

- **Community name and domain** ‚Äî the SSID, local domain, and community identity
- **IPv4 and IPv6 address prefixes** ‚Äî the address space the whole mesh shares
- **AnyCast gateway MAC** ‚Äî the shared virtual MAC address used for gateway routing
- **Mesh SSID and BSSID** ‚Äî the backhaul mesh identifiers (all nodes must match)
- **Country code and regulatory domain** ‚Äî the country setting for WiFi compliance
- **Routing protocol configuration** ‚Äî BMX7 and Babel settings that apply to all nodes
- **Web interface defaults** ‚Äî lime-app port and access policy

## Node-specific settings (do NOT belong in `lime-community`)

These settings vary by node and must not be placed in `lime-community`:

- **Per-node radio channels** ‚Äî directional gateway nodes may need non-default channels
- **Transmit power overrides** ‚Äî some nodes face different environments
- **Physical interface assignments** ‚Äî vary by hardware model
- **Hostname** ‚Äî each node has a unique hostname
- **Per-node IP addresses** ‚Äî LibreMesh auto-generates these; never hardcode them
- **Secrets, passwords, or keys** ‚Äî must never appear in any committed file; see `secrets/README.md`
- **Site-specific contact information** ‚Äî kept in `inventories/sites.yaml`

## How to update a community setting safely

Changing `lime-community` affects **all nodes** and is a **Class D (high-risk) operation** per `rollout-policy.yaml`. The required process is:

1. **Propose the change** in a draft plan. Describe what will change and why.
2. **Review the rollout-policy.yaml** to confirm the change window and canary requirements.
3. **Get explicit approval** from the lead maintainer in the authorized channel.
4. **Apply to the canary ring first** (currently: `lm-associacao-salao`).
5. **Wait for the stabilization period** (24 hours minimum) before promoting.
6. **Apply to the stable ring** after manual canary promotion.
7. **Apply to the trailing ring** last, with human review per node.
8. **Update inventory and log** the change after completion.

Never deploy a change to `lime-community` outside of a declared change window, and never skip the canary stage.

## What NOT to put in this file

To keep the community configuration safe and auditable:

- **No secrets or passwords** ‚Äî not even hashed. Use `secrets/` for anything sensitive.
- **No per-node IP addresses** ‚Äî LibreMesh auto-assigns these from the community prefix.
- **No hardware model-specific settings** ‚Äî use `lime-node` or `node-overrides/` for that.
- **No autogenerated content** ‚Äî `lime-autogen` is read-only and regenerated automatically.
- **No site contact information** ‚Äî that belongs in `inventories/sites.yaml`.
- **No one-off fixes** ‚Äî temporary fixes go in node-level overrides, not the community file.

## Quick reference: where things live

| What you want to change | Where to put it |
|-------------------------|-----------------|
| Community SSID | `lime-community` ‚Üí `config lime 'wifi'` ‚Üí `ap_ssid` |
| Mesh SSID (backhaul) | `lime-community` ‚Üí `config lime 'wifi'` ‚Üí `mesh_ssid` |
| Community IP range | `lime-community` ‚Üí `config lime 'network'` |
| One node's radio channel | `desired-state/mesh/node-overrides/<hostname>.uci` |
| Node hostname | `desired-state/mesh/node-overrides/<hostname>.uci` |
| Firmware version target | `desired-state/mesh/firmware-policy.yaml` |
| Rollout approval rules | `desired-state/mesh/community-profile/rollout-policy.yaml` |
| Physical site info | `inventories/sites.yaml` |
| SSH credentials | `secrets/` (never committed) |
